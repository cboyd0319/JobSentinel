# Harness Effectiveness

JobSentinel does not assign a structural harness score. Presence, correctness,
and effectiveness are separate questions, and file counts or keyword checks do
not prove useful work.

For a harness change, hold the representative task, starting revision, tool
permissions, environment, and evaluator stable where practical. Compare:

- successful completion and verification rate;
- time to the first correct command or useful edit;
- retries, redundant commands, and human interventions;
- false completion claims and review defects;
- startup bytes, state reads, and rediscovery time;
- hook or verification friction;
- fresh-session resumability; and
- harness files touched by routine product work.

Record the first material failure cause: task, instructions, tools, environment,
state, feedback, model limit, or evaluator defect. Keep a control only when it
improves the representative outcome or protects a tested safety boundary. Its
registry row must state when it can retire.
