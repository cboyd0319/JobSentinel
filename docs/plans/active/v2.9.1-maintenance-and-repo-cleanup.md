# v2.9.1 Maintenance And Repo Cleanup

Last updated: 2026-06-22.

## Problem

`v2.9.0` shipped after a large release push. `v2.9.1` should be a small patch
line that cleans the repo, corrects stale state, and fixes confirmed
maintenance issues without expanding product scope.

## Product Direction

The product direction for this patch is maintenance and repo cleanup. Users
benefit when the repo is easier to verify, release claims are accurate, and
regressions are fixed without adding new behavior to learn.

Use these defaults:

- No new product features.
- Prefer deletion, consolidation, and standard repo tooling over new
  abstractions.
- Preserve `v2.9.0` behavior unless fixing a confirmed regression.
- Keep warning, consent, review, and local-first boundaries intact.
- The hard stops remain privacy and security failures, credential or session
  capture, hidden background access, unsafe destinations, data loss, platform
  control bypass, unbounded source access, and external side effects the user
  did not choose.

## Scope And Non-Goals

In scope:

- Active plan cleanup after `v2.9.0` publication.
- Maintained-doc drift cleanup, especially stale pre-publication wording.
- Harness, bloat, docs, security, dependency-pin, and test-quality fixes.
- Deleting disposable generated artifacts or duplicate maintained state.
- Small bug fixes with focused regression coverage.
- Final `v2.9.1` package metadata, changelog, release notes, wiki sync review,
  and release verification when the cleanup checklist is closed.

Non-goals:

- v3 implementation.
- New source adapters, scraping behavior, browser automation, or restricted
  source capabilities.
- New external AI features or provider transports.
- Broad UI redesign.
- Dependency additions unless current tooling is broken and no smaller path is
  available.
- Calling the broad repo-wide quality goal complete.

## Maintenance Checklist

### 1. Active State Cleanup

- Move completed `v2.9.0` release planning out of `docs/plans/active/`.
- Keep active restart files limited to `status.md`, `current-work.md`, and this
  maintenance plan.
- Update `docs/plans/index.json`, `docs/plans/README.md`, `docs/ROADMAP.md`,
  and `docs/harness/manifest.json`.
- Defer v3 implementation from the active workstream index.

### 2. Stale Release Claims

- Scan maintained docs for `v2.9.0` pre-publication language.
- Keep release evidence ledgers when they are clearly historical snapshots.
- Update or move any maintained docs that still say publication, tag movement,
  or hosted asset verification remains active.
- Keep macOS and Windows signing gaps accurate.

### 3. Repo Cleanup

- Delete disposable generated artifacts only after confirming they are not
  tracked release evidence.
- Remove duplicate maintained state when one source of truth is enough.
- Keep machine-specific absolute local paths out of tracked docs and code.
- Keep active docs under harness context budgets.

### 4. Maintenance Fixes

- Accept only confirmed bug, docs, harness, security, privacy, portability, or
  release-tooling fixes.
- Add the smallest focused check that would catch a regression.
- Do not start broad refactors or abstractions without a failing gate or a
  concrete maintenance blocker.

### 5. Release Cut

- Bump package, Tauri, Cargo, lockfile, changelog, and release notes only when
  the patch release is ready to cut.
- Run `npm run release:readiness -- --version 2.9.1` and the focused matrix
  checks for touched surfaces.
- Verify public assets only after the release is published.

## Acceptance Criteria

- Active restart docs and plan index no longer route agents into `v2.9.0`
  publication work.
- v3 implementation is deferred during the patch line.
- Package metadata stays `2.9.0` until a deliberate `v2.9.1` release-cut slice.
- Every cleanup change is backed by focused docs, harness, or regression
  evidence.
- Rule 0 is unchanged.

## Verification

For plan/docs cleanup:

```bash
npm run harness:plan -- --since origin/main
npm run harness:check
npm run lint:docs
npm run lint:bloat
npm run harness:score
git diff --check
```

Broaden only when code, release workflows, security, privacy, packaging, or UI
behavior changes.
