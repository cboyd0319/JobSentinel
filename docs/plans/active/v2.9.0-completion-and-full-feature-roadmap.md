# v2.9.0 Completion And Full-Feature Roadmap

Last updated: 2026-06-19.

## Problem

JobSentinel needs one clear source of truth for what remains before the
`2.9.0` release and what remains before the product can reasonably be called
full featured.

The release and product goals are related but not the same:

- `2.9.0` is close to release-complete. The remaining work is final gate
  execution, release publication, asset verification, and truthful signoff.
- Full-featured JobSentinel still has roadmap work after `2.9.0`: richer
  user-controlled browser assistance, broader source coverage, stronger resume
  and application workflows, deeper job-quality analysis, and lower-friction
  signed distribution.

## Product Direction

JobSentinel's product direction for this goal is maximum job-search help with
minimum user friction. When ordinary implementation guidance conflicts with
making the product easier, more automatic, or more comprehensive for a job
seeker, prefer the path that helps the user find and manage jobs.

Use these defaults:

- Make the easy path automatic when the action is local, reversible, and
  user-reviewed.
- Remove arbitrary roadblocks, redundant confirmations, engineer-only setup,
  and manual copy/paste work.
- Prefer warning, consent, review, and a secure local path over blocking when
  the user is intentionally using a source they control.
- Keep broad public and unauthenticated sources low-friction.
- Keep authenticated or restricted-source work user-initiated, visible, and
  local by default.

The hard stops remain privacy and security failures, credential or session
capture, hidden background access, unsafe destinations, data loss, platform
control bypass, unbounded source access, and external side effects the user did
not choose.

## Scope And Non-Goals

In scope:

- Final `2.9.0` pre-release verification and release operations.
- Remaining source, Browser Import, and restricted-source validation.
- Product capabilities still needed for a mature, full-featured job-search
  assistant.
- Platform, security, privacy, documentation, and user-ease gaps that affect
  release confidence.
- Exact commands and evidence needed before claiming release completion.

Non-goals:

- Capturing or storing cookies, tokens, browser storage, authorization headers,
  or session material.
- Silent background access for authenticated restricted sources.
- Autonomous LinkedIn scraping, scheduled authenticated monitoring, or
  platform-control bypass.
- Claiming Gatekeeper-ready macOS distribution before signing, notarization,
  stapling, and install proof exist.
- Treating every future product idea as a `2.9.0` blocker.

## Constraints And Risks

- Rule 0 controls every milestone: user data stays local unless the user
  explicitly configures an external channel, external AI is optional and
  disabled by default, and sensitive actions require explicit user review.
- Windows 11+, macOS, and Linux behavior must remain intact. If a platform
  cannot be live-checked, the gap must be named.
- Package managers, direct npm packages, npm overrides, Cargo crates, GitHub
  Actions, and build tools must stay exact-pinned to current latest stable where
  upstream constraints allow it.
- Public unauthenticated sources should stay low-friction. Restricted public
  boards can require acknowledgement, rate limits, and clear warnings, but not
  authenticated-session time caps when no sign-in is required.
- Authenticated restricted sources require a plain-language warning and
  explicit user action. JobSentinel can provide the safest local path, but must
  not persist sign-in material or run hidden refreshes.
- Browser Import may import visible, user-chosen data through a user action. It
  must not read hidden page state, browser storage, network traffic, screenshots,
  or authorization material.
- External release blockers are real: Apple Developer ID signing, Windows code
  signing and SmartScreen reputation, source-specific API keys or partner terms,
  and GitHub Actions availability cannot be solved only with local code.

## What Remains For v2.9.0

### 1. Freeze Feature Work

Before release operations start, do not begin another broad feature slice unless
it fixes a failing gate, user-data risk, security issue, source blocker,
documentation drift, or installer problem.

Expected result:

- `main` has a clean local worktree except intentional release metadata.
- The active status, current work plan, release risk register, source-debug
  ledger, verification matrix, README, wiki, and release notes describe the
  same behavior.

### 2. Run Final Local Gates From The Latest Commit

Run the final release gate from the verified commit, currently expected to be
`13f7650a` or a later intentional commit.

```bash
npm run harness:check
npm run release:readiness -- --version 2.9.0
npm run release:check-deps
npm run doctor
npm run lint:docs
npm run lint:bloat
npm run lint:security
npm run lint
npm run test:run
npm run test:e2e:all
npm run build
cd src-tauri && cargo fmt --all -- --check
cd src-tauri && cargo clippy -- -D warnings
cd src-tauri && cargo test --lib
```

Expected result:

- All checks pass from the release commit.
- Any failure is fixed, documented, rechecked, and committed before tagging.

### 3. Recheck Browser Import And LinkedIn-Compatible Paths

Run the focused checks for the newest visible-card import behavior and the
user-controlled LinkedIn-compatible Workbench flow.

```bash
npm run test:run -- src/components/BrowserImportButtonScript.test.ts src/components/LinkedInWorkbench.test.tsx src/components/BookmarkletGenerator.test.tsx
cd src-tauri && cargo test --lib bookmarklet
```

Manual evidence still needed before final release signoff:

- Restricted-source warning copy is plain, friendly, and accurate.
- Browser Import captures only visible user-selected or visible-page job data.
- Imported records remain reviewable before durable use.
- URLs are scrubbed of query strings, fragments, and token-like values before
  storage where required.
- No cookies, storage state, auth headers, HAR, trace, video, or session files
  are produced by authenticated-source testing.

### 4. Confirm Manual UI Coverage

The goal is not complete until every route, click, action, modal, toast, empty
state, loading state, error state, settings panel, import flow, keyboard path,
and narrow-width surface has been exercised and recorded.

Minimum evidence:

- Route inventory screenshots or equivalent Playwright/Computer Use evidence.
- Whole-UI click and keyboard pass.
- Source setup, restricted-source acknowledgement, Browser Import, Workbench,
  reminders, settings, setup wizard, application tracking, resume assistance,
  alerts, data management, privacy, and support surfaces.
- No console errors, broken focus paths, overflow, passive keychain prompts, or
  copy that assumes engineering knowledge.

### 5. Push And Publish Release Assets

Release sequencing:

1. Wait for any older `v2.9.0` workflow run to finish.
2. Push `main` and the public wiki after final local gates pass.
3. Move `v2.9.0` to the verified commit.
4. Let the hosted release workflow build platform assets.
5. Verify the published release, checksums, SBOMs, attestations, skills
   archives, installer names, and install instructions.

Expected result:

- Windows no-account artifacts are published as unsigned where appropriate and
  clearly labeled.
- macOS no-account universal DMG is published with accurate Gatekeeper language.
- Linux AppImage and deb artifacts are published with checksums and SBOMs.
- Skills tar.gz and ZIP archives are published for download.
- The release page, README, docs, wiki, and verification records match.

## Known v2.9.0 External Gaps

These do not need to block a no-account `2.9.0` release if they are documented
truthfully:

- Apple Developer ID signing, notarization, stapling, and Gatekeeper acceptance.
- Windows code-signing certificate and SmartScreen reputation.
- Any source-specific partner API credentials not already configured locally.
- Public hosted workflow failures caused by external service outage rather than
  repo logic.

## What Remains To Be Full Featured

### User-Controlled Browser Assistance

JobSentinel should become more helpful during human-controlled browser sessions
without becoming a hidden scraper.

Needed capabilities:

- Visible local controls such as `Log applied`, `Track this job`, `Save locally`,
  `Add note`, and `Import visible jobs`.
- Local activity ledger entries for applied, saved, rejected, interview,
  follow-up, reminder, and note events.
- Optional interest learning from JobSentinel-side actions such as saved jobs,
  dismissed jobs, search terms, notes, imported visible details, and user
  feedback.
- Clear on/off state for any future watch-and-learn mode, with exact copy that
  says what is recorded and what is not.
- No durable record created without user confirmation.

### Source Expansion And Discovery

JobSentinel should help users find jobs beyond the stock boards.

Needed capabilities:

- Stronger native adapters or reviewed source definitions for Greenhouse, Lever,
  Ashby, SmartRecruiters, Workday, Workable, Recruitee, Breezy, JazzHR,
  Bullhorn, company career pages, and regional boards.
- Source classification that distinguishes public API, public unauthenticated
  HTML, restricted public board, authenticated restricted session, and unknown
  review-required source.
- Employer career-page discovery for companies that do not reliably syndicate
  to broad job boards.
- Regional and country-specific source coverage for the United States, United
  Kingdom, India, Canada, and other user-requested markets.
- Continued source-debug fixtures, low-volume live probes, and documented
  source-specific reasons for restricted-domain records.

### Resume, Matching, And Application Assistance

Resume assistance should be useful for technical and non-technical users.

Needed capabilities:

- Stronger hard-requirement checks with evidence caps and fail-fast guidance.
- Better ATS and readability review that never invents unsupported experience.
- Role-family coverage for technical, content, operations, healthcare, service,
  trades, education, sales, and early-career job seekers.
- Form-review provenance for exact questions, user-confirmed answers, voluntary
  or protected questions, and unknowns that require user confirmation.
- Safer attachment, cover letter, and screening-answer workflows.

### Job Quality And Ghost-Job Review

JobSentinel should help users spend time on better opportunities.

Needed capabilities:

- Duplicate and repost clustering across sources.
- Public employer or ATS follow-through checks where allowed.
- Freshness, low-detail, vague-role, stale-posting, scam-cue, and pay-risk
  signals with confidence levels.
- User feedback loops that improve future scoring without claiming employer
  intent as fact.
- Clear language that distinguishes poor fit, low confidence, scam evidence,
  and likely ghost-job indicators.

### Guided Search Operations

JobSentinel should guide the job search week by week.

Needed capabilities:

- Intake that keeps broad defaults and does not silently narrow the user.
- Weekly review, stop rules for weak lanes, stale application thresholds, and
  replan-from-tracker workflows.
- Next-best-action suggestions that are always reviewable and explainable.
- Referral, recruiter, outreach, follow-up, interview-prep, and offer-review
  handoffs across the downloadable skills and product UI.

### Pay, Offer, And Risk Review

JobSentinel should protect users from weak pay information and rushed offers.

Needed capabilities:

- Written-vs-verbal offer separation.
- Total compensation comparison, commute and relocation cost review, deadline
  pressure checks, and counter or decline templates.
- Pay transparency support by region where data and law are current.
- Plain review of missing, minimum-only, maximum-only, malformed, broad, or
  anchoring-prone listed pay.

### Distribution And Support Polish

The app should be easy for non-technical users to install, run, update, and
recover.

Needed capabilities:

- Signed Windows and macOS installers when credentials exist.
- Auto-update path with integrity checks and rollback guidance.
- Backup, restore, export, import, recovery, and support-report flows that do
  not leak sensitive data.
- Beginner-readable install, first-run, privacy, source, and troubleshooting
  docs across README, docs, wiki, and release notes.
- Continued accessibility, keyboard, screen-reader, narrow-width, empty-state,
  loading-state, error-state, and platform parity checks.

## Progress

- Current local state: `2.9.0` is close to release-complete, with release
  operations and final verification remaining.
- Current product state: JobSentinel has the secure local-first foundation,
  expanded skills, Browser Import, source taxonomy, restricted-source policy,
  encrypted storage, and broad verification evidence, but is not done as a
  mature full-featured product.
- 2026-06-19 progress: release gates found dependency drift and updated
  `@tauri-apps/cli` to `2.11.3`; source recognition expanded Application
  Assist ATS detection for additional career systems already listed in the
  shared source taxonomy without adding new scraper transport or submit
  automation.
- Next best action: run the final local release gate from the latest commit,
  then push, tag, build, publish, and verify release assets.

## Decision Log

- Keep `2.9.0` focused on releasing the verified secure foundation rather than
  absorbing every future source and browser-assistance idea.
- Keep public unauthenticated sources low-friction where source behavior allows
  it.
- Keep authenticated restricted sources user-controlled, warning-gated, and
  local, without token or session storage.
- Use Browser Import and local activity ledgers as the bridge between helpful
  automation and privacy-preserving user control.
- Treat signed distribution as an upgrade path when credentials exist, not as a
  blocker to the no-account release path.
