#### 1.1 Wire UndoContext to Dashboard Actions

**Problem:** UndoContext exists with full undo/redo stack and Ctrl+Z support, but no
actions actually use it.

**Files to modify:**

- `src/pages/Dashboard.tsx` - Import useUndo, wrap actions
- `src/hooks/useJobOperations.ts` - Add undo wrappers

**Implementation:**

```typescript
// In useJobOperations.ts
const { pushAction } = useUndo();

const handleHideJob = async (jobId: string) => {
  await invoke("hide_job", { jobId });
  pushAction({
    type: "hide",
    description: "Job hidden",
    undo: async () => await invoke("unhide_job", { jobId }),
    redo: async () => await invoke("hide_job", { jobId }),
  });
  refreshJobs();
};
```

**Backend requirement:** Add `unhide_job` command if not exists.

**Tests:**

- Unit test: pushAction is called on hide
- Integration test: Ctrl+Z restores hidden job

#### 1.3 Auto-Detect Location

**Problem:** Users must manually enter city/state. We can auto-detect from IP.

**Files to create:**

- `src-tauri/src/core/location/mod.rs` - Geolocation module

**Files to modify:**

- `src-tauri/src/core/mod.rs` - Add location module
- `src-tauri/src/commands/mod.rs` - Add detect_location command
- `src/pages/SetupWizard.tsx` - Call on mount, pre-fill fields
- `src/pages/Settings.tsx` - Add "Detect Location" button

**API choice:** `ip-api.com` (free, no API key, 45 req/min limit)

**Implementation:**

```rust
// src-tauri/src/core/location/mod.rs
#[derive(Serialize, Deserialize)]
pub struct LocationInfo {
    pub city: String,
    pub region: String,  // State
    pub country: String,
    pub country_code: String,
}

pub async fn detect_location() -> Result<LocationInfo> {
    let resp = reqwest::get("http://ip-api.com/json/")
        .await?
        .json::<serde_json::Value>()
        .await?;
    
    Ok(LocationInfo {
        city: resp["city"].as_str().unwrap_or("").to_string(),
        region: resp["regionName"].as_str().unwrap_or("").to_string(),
        country: resp["country"].as_str().unwrap_or("").to_string(),
        country_code: resp["countryCode"].as_str().unwrap_or("").to_string(),
    })
}
```

**Tests:**

- Unit test: Parse valid response
- Unit test: Handle network error gracefully

### Phase 4: Profile Import (Priority 4)

#### 4.1 LinkedIn Profile Import

**Problem:** Users manually enter all profile info. LinkedIn has it already.

**Files to create:**

- `src-tauri/src/core/linkedin/profile.rs`

**Approach:**

- Accept public LinkedIn profile URL
- Scrape public HTML (no auth required for public profiles)
- Extract: name, headline, skills, experience titles
- Handle blocking gracefully (show manual entry fallback)

**Risk:** LinkedIn actively blocks scrapers. This is best-effort.

**Alternative:** Encourage JSON Resume import instead.

---

#### 4.2 Smart Screening Answers

**Problem:** Users re-enter same screening answers (work auth, start date) every time.

**Database schema:**

```sql
CREATE TABLE screening_answer_history (
  id INTEGER PRIMARY KEY,
  question_pattern TEXT NOT NULL,  -- Regex or keyword match
  answer_text TEXT NOT NULL,
  times_used INTEGER DEFAULT 1,
  last_used_at TEXT,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP
);
```

**Implementation:**

- After successful apply, save Q&A pairs
- On new application, pattern-match questions
- Pre-fill with most-used answer
- Show "Use saved answer" suggestion
